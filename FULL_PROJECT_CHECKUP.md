# Полный чек-ап проекта TG Music Player

## Краткое резюме

Проект рабочий и уже содержит значительный объём функциональности:
- Telegram Mini App frontend на React + Vite
- FastAPI backend
- поиск треков через Hitmo parser
- радио
- плейлисты
- избранное
- скачивание в приложение и в Telegram
- подписки / рефералка / админка / тексты / YouTube

Но при этом в проекте накопился заметный технический долг. Основные проблемы не в одной крупной поломке, а в сочетании факторов:
- слишком большой и перегруженный `App.tsx`
- дублирование логики между frontend-компонентами
- смешение бизнес-логики, UI и побочных эффектов
- слабая консистентность между frontend и backend моделями
- рискованные настройки production/deploy/security
- следы экспериментов и старых веток реализации внутри репозитория

Ниже — детальный аудит.

## 2. Критичные логические проблемы

## 2.1. `App.tsx` перегружен и содержит слишком много ответственности

### Что не так
`App.tsx` очень большой (`~2000+` строк) и совмещает в себе:
- роутинг между вкладками
- поисковую логику
- UI home/radio/playlists/library/admin/subscription
- модалки
- player UI
- lyrics UI
- referral UI
- YouTube UI
- playlist editing
- radio loading
- pagination
- toast/feedback

### Почему это проблема
- очень высокий риск регрессий
- трудно находить причину багов
- одна правка может ломать несвязанный экран
- сложно тестировать
- сложно делегировать работу по файлу

### Приоритет
**Высокий**

### Рекомендация
Разделить на:
- `views/HomeView.tsx`
- `views/RadioView.tsx`
- `views/PlaylistsView.tsx`
- `views/LibraryView.tsx`
- `components/player/*`
- `hooks/useSearch.ts`
- `hooks/useRadio.ts`
- `hooks/useLyrics.ts`

---

## 2.2. Есть дублирующиеся фронтенд-деревья и мёртвый код

### Что найдено
- существует `App.tsx` и параллельно `newfront/`
- существует `views/HomeView.tsx`, но фактически главный Home рендерится внутри `App.tsx`
- часть логики продублирована между `App.tsx` и `HomeView.tsx`

### Почему это проблема
- легко исправить не тот файл
- повышенный шанс «кажется исправил, но ничего не поменялось»
- аудит и поддержка усложняются

### Приоритет
**Высокий**

### Рекомендация
- удалить или архивировать `newfront/`, если он не используется
- удалить/переписать `HomeView.tsx`, если он неактивен
- оставить одну каноническую реализацию на экран

---

## 2.3. Расхождение логики поиска между `App.tsx`, `PlayerContext.tsx` и `HomeView.tsx`

### Что найдено
Поисковая логика размазана по нескольким местам:
- `App.tsx` делает debounce search и load more
- `PlayerContext.tsx` тоже содержит `loadMoreTracks`
- `HomeView.tsx` содержит собственный search debounce и pagination

### Почему это проблема
- разные экраны могут вести себя по-разному
- возможны конфликты состояний и дубли запросов
- трудно понять, какой код реально активен

### Приоритет
**Высокий**

### Рекомендация
Вынести единый search flow в одно место:
- либо в `PlayerContext`
- либо в отдельный `useSearch` hook

---

## 2.4. Недавний баг с radio показал хрупкость текущего рендера

### Что найдено
Ранее radio-строки попадали в Home search area из-за переиспользования DOM между view.

### Почему это важно
Это симптом того, что:
- рендер-структура слишком монолитная
- view не изолированы должным образом
- React reconciliation легко ломается при похожих корневых структурах

### Статус
Исправлено через `key` на корневых view.

### Рекомендация
Среднесрочно — вынести экраны в отдельные компоненты, а не держать их как большие inline render functions.

## 11. Итоговая оценка

### Сильные стороны
- Проект уже функционально насыщен
- Есть живая backend/frontend связка
- Есть оффлайн-хранилище и полезные пользовательские функции
- Есть основа для монетизации и админки

### Слабые стороны
- Слишком много логики сконцентрировано в нескольких больших файлах
- Репозиторий содержит legacy/служебный шум
- Безопасность и production hygiene требуют срочного внимания
- Документация не отражает реальное состояние проекта

### Общий вывод
Проект **не выглядит безнадёжным** — наоборот, он уже достаточно зрелый по функциям. Но сейчас он находится в стадии, когда без структурного наведения порядка каждое следующее изменение будет всё дороже и рискованнее.

Главный вывод аудита:
**сейчас важнее не добавлять ещё больше функций, а стабилизировать архитектуру, безопасность и консистентность кода.**
